### 数据 schema

生成的 parquet 保持 CoSearch/VERL `RLHFDataset` 需要的结构，主要字段包括：

- `data_source`
- `prompt`
- `ability`
- `reward_model`
- `extra_info`

其中 `extra_info["question"]` 是 `CoSearchAgentLoop` 需要的关键字段。

### Prompt 格式

生成数据时使用 Search-R1 风格工具调用格式，核心要求：

- 每轮 assistant 输出 `<reason>...</reason>`。
- 然后输出 `<tool_call>...</tool_call>` 或 `<answer>...</answer>`。
- 第一轮必须调用 search。
- 未收到工具结果前不允许直接输出 answer。
- `<tool_call>` 内必须是 JSON：

```json
{
  "name": "search",
  "arguments": {
    "query": "<string>"
  }
}
```

这个 prompt 是为了降低 Qwen3-0.6B 替代基模在 XML/JSON 工具格式上的失败率。

### 为什么写 03 和 05 两层脚本

`train_cosearch_verl_base.sh` 是更通用的官方 VERL 参数包装器，里面集中维护：

- Ray 参数
- trainer 参数
- actor rollout 参数
- reranker rollout 参数
- FSDP/LoRA 参数
- reward 参数
- CoSearch tool config
- agent loop config
- GPU 切分

`05_train_paper_qwen3_dense_smoke.sh` 是 paper-path smoke 默认入口，里面设置适合当前服务器显存的默认值：

- 默认 `GPU_IDS=0,1,2,3`
- 默认 `TOP_N=50`
- 默认 `TOP_M=5`
- 默认 `MAX_PROMPT_LENGTH=11264`
- 默认 `MAX_RESPONSE_LENGTH=1024`
- 默认 `MAX_MODEL_LEN=12288`
- 默认 `MAX_NUM_SEQS=1`
- 默认 `GPU_MEMORY_UTILIZATION=0.20`
- 默认 `TOTAL_STEPS=1`
- 默认 `TRAIN_MAX_SAMPLES=4`

这样后续扩展时可以只改环境变量，不需要直接改官方 trainer 参数。

### GPU 切分

官方 CoSearch 需要 main agent 和 reranker 两套 actor/rollout 资源。当前脚本要求至少 2 张 GPU。

成功 smoke 使用：

```text

```

脚本会把 visible GPUs 交给 VERL/Ray，由官方 resource pool 分配 main agent 和 reranker。

为了避免 OOM，当前 smoke 默认压缩了：

- `TRAIN_MAX_SAMPLES`
- `VAL_MAX_SAMPLES`
- `TRAIN_BATCH_SIZE`
- `VAL_BATCH_SIZE`
- `MAX_NUM_SEQS`
- `GPU_MEMORY_UTILIZATION`

但保留了 paper-path 的关键逻辑：

- dense retrieval top-50
- reranker top-5
- tool-call agent loop

- checkpoint save/load

## 官方代码补丁详情

### Ray faulthandler 补丁

文件：

```text
CoSearch/main_co_search_ppo.py
```

问题：

Ray 在当前服务器上初始化时可能尝试安装 faulthandler，但因为系统资源限制出现：

```text
OSError: [Errno 12] Cannot allocate memory
```

处理：

- 在 `ray.init` 前 monkeypatch Ray 内部 faulthandler 设置。
- 只捕获 errno 12。
- 继续执行 Ray 初始化。

影响：

- 不改变训练算法。
- 不改变 rollout。
- 不改变 reward。
- 只是跳过 Ray 的调试/崩溃栈相关 faulthandler 设置。

### Qwen3 tool-call 解析补丁

文件：

```text
CoSearch/verl/verl/experimental/agent_loop/co_search_agent_loop.py
```

问题：

Qwen3-0.6B 经常在一轮输出里同时出现类似：

```text
<tool_call>...</tool_call>
<answer>...</answer>
```

官方逻辑容易提前把 answer 当成最终答案，导致工具调用没有执行，最终 valid rate 很低。

处理：

- 先检查并解析合法 `<tool_call>`。
- 如果存在合法 tool call，就先执行 search。
- 对输出进行截断，只保留第一个 `</tool_call>` 之前的 token。

影响：

- 更适配 Qwen3-0.6B。
- 不改变 CoSearch 的工具调用机制。
- 不改 reward 逻辑。

### reranker metric 加权平均补丁

文件：

```text
CoSearch/verl/verl/experimental/agent_loop/agent_loop.py
```

问题：

Qwen3-0.6B 初始阶段 reranker 格式不稳定。当所有 reranker 分组权重为 0 时，官方 metric 聚合里 `np.average(..., weights=...)` 会出现除零错误。

处理：

- 增加 `safe_weighted_average`。
- 当权重为空或总和为 0 时返回普通均值或 0。

影响：

- 只影响日志 metric 聚合。
- 不改变训练样本生成。
- 不改变 reward 计算主路径。

### reranker enable_thinking 补丁

文件：

```text
CoSearch/verl/verl/tools/co_search_tool.py
```

问题：

Qwen3 chat template 默认可能启用 thinking 风格输出，容易干扰 CoSearch reranker 的结构化输出格式。

处理：

```text
enable_thinking=False
```

影响：

- 减少 `<think>` 干扰。
- 保持 reranker prompt 和 scoring 逻辑不变。

## 为什么当前指标低

当前 smoke 指标为负，并不是流程失败，而是由以下原因造成：

1. 论文基模被替换为 `Qwen3-0.6B`，模型能力和格式遵循能力明显弱于论文设置。
2. 只训练了 1 个 PPO step。
3. 训练只用了 4 条样本。
4. 评测只用了 8 条样本。
5. Qwen3-0.6B 初期经常输出不符合 Search-R1 XML/JSON 规范的内容。
6. reranker 也存在格式失败，触发 fallback。
7. CoSearch reward 中格式错误会被惩罚为 `-0.2`。

所以当前分数的意义是：

```text
官方训练/评测链路已经真实跑通。
```

不是：

```text
模型已经达到论文分数。
```

## 踩坑记录

### 1. HuggingFace 不可用

服务器网络无法稳定访问 HuggingFace，所以不要浪费时间反复尝试 HF 下载。

处理方式：

- Qwen3 使用本地已有模型目录。
- E5 使用 ModelScope 或本地已有目录。
- 数据使用本地 FlashRAG 数据。

### 2. Ray faulthandler ENOMEM

现象：

```text
OSError: [Errno 12] Cannot allocate memory
```

发生在 Ray 初始化阶段，不是模型 OOM。

处理：

- 对 Ray faulthandler 设置做兼容补丁。

### 3. vLLM 上下文长度不足

早期较短上下文会导致 reranker prompt 或 tool response 被截断。最后成功 smoke 使用：

```text
MAX_PROMPT_LENGTH=11264
MAX_RESPONSE_LENGTH=1024
MAX_MODEL_LEN=12288
MAX_TOOL_RESPONSE_LENGTH=2048
MAX_NUM_SEQS=1
```

注意：

- async vLLM 实际需要容纳 prompt + response。
- 不能只看 `MAX_MODEL_LEN`，还要看 `MAX_PROMPT_LENGTH` 和 `MAX_RESPONSE_LENGTH` 的组合。

### 4. Qwen3 输出格式不稳定

Qwen3-0.6B 会出现：

- 同一轮同时输出 tool call 和 answer。
- reranker 输出带 `<think>`。
- JSON 不完整。
- answer 过早出现。

处理：

- prompt 强化第一轮必须 search。
- chat template 设置 `enable_thinking=False`。
- agent loop 优先执行合法 tool call。
- 截断第一个完整 tool call 之后的输出。

### 5. reranker metric 权重为 0

早期因为 reranker 格式失败，metric 聚合可能除零。

处理：

- 增加安全加权平均。

### 6. GPU 显存控制

最开始尝试较大上下文或较多序列时容易 OOM。成功 smoke 使用 4 张 GPU，并降低：

- batch size
- rollout 数
- max num seqs
- GPU memory utilization
- agent workers

后续扩大训练时应逐步增加，不要一次性拉满。
