### 3. 报告生成脚本：`11_summarize_qwen3_4b_probe.py`

路径：

```text
scripts/cosearch_local/11_summarize_qwen3_4b_probe.py
```

作用：

- 从训练日志中解析 `step:<id> - ...` 形式的 metrics。
- 重点解析：
  - `timing_s/step`

- 从 GPU 采样 CSV 读取 `nvidia-smi` 观察到的峰值显存和利用率。
- 估算 51,200 条训练集 1 epoch 需要多少 step 和多少小时。
- 输出 JSON、追加 CSV、生成 Markdown 报告。

注意：如果直接运行 `09`，不会自动调用这个汇总脚本；如果通过 `12` 跑 sweep，则会自动调用。

## GPU 划分策略

当前不是主 Agent 和 reranker 共享同一组 8 卡，而是官方 dual-agent trainer 默认做资源池拆分。

在 `CoSearch/main_co_search_ppo.py` 中，单机模式下：

```text
trainer.n_gpus_per_node = 8
main_agent_pool = 前 4 张 GPU
reranker_pool = 后 4 张 GPU
```

日志中会出现类似：

```text
[Resource Split] Single node mode: 8 GPUs
[Resource Split] Main agent: GPUs 0-3
[Resource Split] Reranker: GPUs 4-7
```

角色映射：

```text
main_agent_pool:
  Role.ActorRollout
  Role.RefPolicy

reranker_pool:
  Role.RerankerActorRollout
  Role.RerankerRefPolicy
```

是否能改成共享 8 卡：

- 理论上可以，但需要改官方 trainer 的 resource pool 映射。
- 不是简单改 shell 参数。
- 共享池会让主 Agent、reranker、ref、rollout worker 争抢同一批 GPU。
- 在 `Qwen3-4B + 12288 context + N_ROLLOUTS=8` 下，共享 8 卡可能更容易 OOM 或调度不稳定。
- 当前建议先用官方 4+4 split 探出最快稳定配置，再单独做 shared global pool 分支对比。

## Dense retriever 相关修复

### 问题

一开始直接运行 `09` 时，训练过程中出现大量：

```text
Retrieval failed after 1 attempts:
Search API call failed:
Cannot connect to host 127.0.0.1:8010
```

原因是旧版 `09` 只设置：

```text
RETRIEVAL_SERVICE_URL=http://127.0.0.1:8010/retrieve
```

但没有启动 dense retriever。直接运行 `09` 时，`8010` 没有服务监听，所以 worker 连不上检索 API。

### 修复

已在 `09` 中新增：

- `check_retriever`
- `ensure_retriever`
- `cleanup_retriever`

现在 `09` 启动流程是：

1. 先请求 `http://127.0.0.1:${PORT}/retrieve` 做健康检查。
2. 如果服务已存在，直接复用。
3. 如果不存在，且 `START_RETRIEVER` 不是 `0/false/no`，自动启动：

```text
scripts/cosearch_local/02b_start_dense_retriever_server.sh
```

4. 等健康检查通过后才进入训练。
5. 如果 retriever 是 `09` 自己启动的，脚本退出时自动关闭。

### 与 `12` 的关系

`12_run_qwen3_4b_paper_down_sweep.sh` 自己会启动 retriever，所以它调用 `09` 时传入：

```text
START_RETRIEVER=0
```

避免重复启动。


这表示当时还没有完成第一个完整 step，因此日志里暂时没有 step 耗时指标。

### 2026-06-03 最新排查结论

`10.30.log` 后段的实际退出原因不是原先的训练 step 内 OOM，而是 vLLM 初始化时的 allocator 断言：

```text
AssertionError: Expandable segments are not compatible with memory pool.
```

触发原因：

- `train_cosearch_verl_base.sh` 曾默认设置 `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True`。
- 该环境变量会被 Ray/vLLM worker 继承。
- 当前 vLLM 使用 `CuMemAllocator` memory pool，代码中明确断言配置不能包含 `expandable_segments:True`。

已修复：

- `train_cosearch_verl_base.sh` 不再默认设置 `expandable_segments:True`。
- 如果外部环境已经带了 `PYTORCH_CUDA_ALLOC_CONF=...expandable_segments:True...`，脚本会提示 warning 并清掉该变量。
- 如确实要强制保留该不兼容配置，需要显式设置 `ALLOW_VLLM_EXPANDABLE_SEGMENTS=1`，但当前不建议这样做。

日志中的 Flash Attention 2 dtype warning 不是这次退出的直接原因。后续配置打印里可以看到模型配置包含：

```text
"attn_implementation": "flash_attention_2"
"dtype": "bfloat16"
```

## 常见 warning 解释

### 1. Reranker 编号重复

日志示例：

```text
Reranker output format invalid: ['<rerank> indices must be unique.'].
Falling back to retrieval top-5. Raw output: N/A
```

意思：

- reranker 被调用了。
- 但它生成的 `<rerank>` 结果中出现重复编号。
- CoSearch 要求 reranker 输出 5 个互不重复的文档编号。
- 因此该次 reranker 输出无效。
- 系统回退到 dense retriever 的 top-5。

期望格式类似：

```text
<reason>...</reason>
<rerank>[3] > [12] > [1] > [7] > [20]</rerank>
```

错误输出可能类似：

```text
<reason>...</reason>
<rerank>[3] > [3] > [7] > [10] > [12]</rerank>
```

### 2. 缺少 `<reason>` 或 `<rerank>`

日志示例：

```text
Reranker output format invalid:
['Missing <reason> block.',
 'Tail after </reason> must contain only one <rerank>...</rerank> block (no <select>, no extra text).',
 'Missing <rerank> block in tail.'].
Falling back to retrieval top-5. Raw output: N/A
```

意思：

- reranker 输出没有符合 CoSearch 规定的结构。
- 可能没有 `<reason>...</reason>`。
- 可能没有 `<rerank>...</rerank>`。
- 可能在 `</reason>` 后写了额外自然语言、`<select>` 标签，或其他不允许内容。

正确格式必须类似：

```text
<reason>The documents are ranked by how directly they answer the query.</reason>
<rerank>[4] > [1] > [8] > [2] > [13]</rerank>
```

影响：

- 训练不会中断。
- 该次工具调用会回退到 dense retrieval top-5。
- reranker metrics 中会记录 fallback。
- `tool_score` 会使用 format penalty，当前默认 `-0.2`。

原因：

- 当前使用 `Qwen3-4B` 替代论文中的 Qwen2.5 instruct 模型。
- 训练早期 reranker 还没有学会严格遵守 CoSearch 输出协议。
- 日志中显示使用的是 training sampling params，温度通常为 `1.0`，输出随机性较高。
- 因此早期出现格式错误是预期现象，但如果比例过高，会影响 reranker 学习效率。

## 当前建议

1. 如果只是验证单组配置是否能跑，直接用 `09`。
2. 如果要系统性比较速度和显存，用 `12`。
3. 如果直接跑 `09`，建议用 `tee` 保存日志，否则只有 terminal 输出。
4. 如果关注 step 耗时，必须等至少一个完整 step 结束后再查 `timing_s/step`。
5. 如果 reranker format warning 很多，先不要急着判定失败；这是训练早期常见现象。但如果 5 step 内几乎全 fallback，则说明 Qwen3-4B 在当前 prompt/采样参数下格式遵循很差，需要考虑：
   - 降低 reranker sampling temperature。
   - 在 reranker prompt 中加强格式约束。
   - 使用更接近论文的 instruct 模型。
   - 先做 SFT/格式 warmup，再做 RL。
6. 当前不要随意改 `N_ROLLOUTS` 和上下文长度，因为这是本轮实验的固定约束。
