# CoSearch FAQ - 250604

## Q1. 当前 CoSearch 的训练计算了哪些 reward？分别是怎么设计的？

当前本地 09/09b 训练走的是官方 CoSearch/VERL dual-agent 训练路径，底层入口是 `scripts/cosearch_local/train_cosearch_verl_base.sh`。该脚本显式设置：

- `reward_model.reward_manager=multiturn`
- `reward_model.use_reward_loop=True`
- `algorithm.use_kl_in_reward=False`
- `custom_reward_function.path=CoSearch/verl/verl/utils/reward_score/search_qa_f1_with_format_penalty.py`
- `custom_reward_function.name=search_qa_f1_penalty_compute_score`

因此本次训练里 reward 主要分两类：main agent reward 和 reranker reward。

### 1. Main Agent Reward

main agent 的 reward 用于训练主 CoSearch agent，也就是负责多轮搜索、调用 retrieval tool、最后输出答案的模型。

实现位置：

- `CoSearch/verl/verl/utils/reward_score/search_qa_f1_with_format_penalty.py`
- 函数：`search_qa_f1_penalty_compute_score`

设计逻辑：

1. 从最终输出中抽取最后一个 `<answer>...</answer>`。
2. 用标准化后的 token-level F1 对比 ground-truth answer，得到 `ans_score`，范围通常是 `0~1`。
3. 检查多轮输出格式是否正确：
   - 中间工具调用轮需要 `<reason>...</reason><tool_call>...</tool_call>`
   - 最终回答轮需要 `<reason>...</reason><answer>...</answer>`
4. 如果格式正确，最终 reward = `ans_score`。
5. 如果格式错误，最终 reward = `format_penalty`，当前默认是 `-0.2`。

所以 main agent 的 reward 可以理解为：

```text
main_reward = F1(answer, ground_truth)          if format valid
main_reward = -0.2                             if format invalid
```

它不是传统分类 loss，也不是纯 EM。它是带格式惩罚的 QA F1 reward。

### 2. Reranker Tool Reward

reranker 在 CoSearch 中被当成一个可训练的 tool/agent，用于对 retrieval 得到的文档进行选择或重排。reranker 自己会先产生一个 tool 输出，然后系统用这个输出继续让 main/search-r1 agent 回答问题。

reranker tool reward 的主要来源是 reranker 输出是否成功、以及它选择/重排出的文档对最终回答是否有帮助。当前实际 dual loop 中，reranker 工具调用的打分由 `call_tool_as_agent` 返回 `tool_reward` 与 `tool_metric`。

相关实现位置：

- `CoSearch/verl/verl/experimental/agent_loop/search_r1_dual_agent_loop.py`
- `_call_reranker_tool_as_agent`
- `aggregate_tool_and_agent_reward`

关键逻辑：

1. 如果 reranker 输出格式校验失败，`tool_reward = -0.2`。
2. 如果 reranker 调用崩溃或执行错误，会标记为无效样本，训练数据构造时会过滤。
3. 如果 reranker 成功，则保留其 `tool_reward`，这个 reward 反映 reranker 选择文档的质量。

代码库里也保留了独立 reranker reward manager，可计算 MRR、Recall、Average Hit Rate 等 ranking reward：

- `CoSearch/verl/verl/workers/reward_manager/reranker_reward_manager.py`
- `CoSearch/verl/verl/utils/reward_score/reranker_reward_functions.py`

但当前 09/09b dual-agent 路径中，训练日志里的 reranker reward 不是单独调用这个 manager 重新算出来的，而是在 agent loop 阶段写进 `rm_scores`，PPO 阶段直接读取。

### 3. Reranker Final Reward

reranker 最终用于 PPO 的 reward 不是简单等于 tool reward，而是把 reranker tool reward 和后续 main/search-r1 agent 的回答 reward 聚合。

当前实际聚合公式在 `search_r1_dual_agent_loop.py::aggregate_tool_and_agent_reward`：

```text
if reranker failed:
    reranker_final_reward = -0.2
else:
    binary_agent_reward = 1.0 if agent_reward >= 0.8 else 0.0
    reranker_final_reward = max(tool_reward, binary_agent_reward)
```

含义：

- 如果 reranker 格式不对，直接给格式惩罚 `-0.2`。
- 如果 reranker 成功，并且后续 agent 在使用 reranker 结果后答案质量足够好，也就是 `agent_reward >= 0.8`，则给 reranker 一个二值成功奖励 `1.0`。
- 否则使用 reranker 自己的 `tool_reward`。

这体现了 CoSearch 的核心设计：reranker 不只是为“排序指标”优化，也为“是否帮助下游 agent 回答正确”优化。

### 4. PPO 阶段实际使用哪个 reward？

Agent loop 会把每条样本的最终 reward 写入 `rm_scores`，位置在 response 的最后一个有效 token 上。PPO 阶段在 `process_single_agent_ppo_step` 中直接读取：

```text
reward_tensor = batch.batch["rm_scores"]
batch.batch["token_level_scores"] = reward_tensor
```

因为当前脚本设置 `algorithm.use_kl_in_reward=False`，所以：

```text
token_level_rewards = token_level_scores
```

如果以后打开 `algorithm.use_kl_in_reward=True`，则 reward 会再扣 KL penalty，这时 `token_level_rewards` 和 `token_level_scores` 就不一定相等。

## Q2. `training_metrics_report.md` 里的指标分别是什么含义？重点解释 `main/score_mean`、`reranker/score_mean`、`main_critic/rewards/mean`、`reranker_critic/rewards/mean`

`training_metrics_report.md` 是从当前 run 的 `*.metrics.jsonl` 汇总出来的。`metrics.jsonl` 每个 train step 一条 JSON，包含 main agent、reranker agent、PPO、reward、长度、资源和耗时指标。

### 重点指标

#### `main/score_mean`

这是 main agent 在当前 step rollout batch 上的平均 sequence score。

计算方式：

```text
main/score_mean = mean(sum(token_level_scores per sample))
```

在当前配置下，main agent 的 `token_level_scores` 就是前面说的带格式惩罚 QA F1 reward：

- 格式正确：score = 最终答案与 ground truth 的 F1
- 格式错误：score = `-0.2`

因此 `main/score_mean` 可以粗略理解为“当前 step 主 agent 平均回答质量”，但它受到格式惩罚影响，不是纯 F1。

#### `main/f1_mean`

这是 reward extra info 里记录的纯答案 F1 均值。

它只反映 `<answer>` 抽出的答案和 ground truth 的 F1，不直接等于最终 reward。区别是：

- `main/f1_mean`：答案内容匹配程度
- `main/score_mean`：格式正确时等于 F1，格式错误时变成 `-0.2`

如果格式错误较多，`main/score_mean` 会明显低于 `main/f1_mean`。

#### `main/valid_rate`

这是 main agent 输出格式有效的比例。

计算来源是 reward extra info 里的 `valid` 字段：

```text
main/valid_rate = count(valid == True) / total
```

它衡量模型是否遵守 CoSearch 多轮格式，例如 `<reason>`、`<tool_call>`、`<answer>` 的结构是否正确。

#### `main_critic/rewards/mean`

这是 PPO 数据指标里 main agent 的平均 sequence reward。

计算方式：

```text
main_critic/rewards/mean = mean(sum(token_level_rewards per sample))
```

当前脚本设置 `algorithm.use_kl_in_reward=False`，所以：

```text
token_level_rewards = token_level_scores
```

因此在这次训练日志里：

```text
main_critic/rewards/mean == main/score_mean
```

如果以后打开 KL-in-reward，`main_critic/rewards/mean` 会变成“score 扣除 KL penalty 后的训练 reward”，届时它可能低于 `main/score_mean`。

#### `reranker/score_mean`

这是 reranker agent 在当前 step reranker training batch 上的平均 sequence score。

计算方式同样是：

```text
reranker/score_mean = mean(sum(token_level_scores per reranker sample))
```

但它的含义和 main 不同。reranker 的 score 是聚合后的 reranker final reward：

```text
if reranker format failed:
    score = -0.2
else:
    score = max(tool_reward, 1.0 if downstream_agent_reward >= 0.8 else 0.0)
```

所以 `reranker/score_mean` 衡量的是 reranker 对下游回答的帮助程度和自身工具输出质量，而不是最终 QA 的纯 F1。

#### `reranker_critic/rewards/mean`

这是 PPO 数据指标里 reranker agent 的平均 sequence reward。

当前同样因为 `algorithm.use_kl_in_reward=False`：

```text
reranker_critic/rewards/mean == reranker/score_mean
```

如果以后打开 KL-in-reward，它会变成扣除 KL penalty 后实际用于 PPO advantage/return 的 reward。

### PPO / loss 类指标

#### `main_actor/pg_loss`、`reranker_actor/pg_loss`

PPO policy-gradient loss。它是策略更新的优化目标之一。数值接近 0 不代表模型没有训练，因为 PPO 的 advantage 归一化、clip、batch 采样都会让这个值很小。看它时要结合 reward、KL、entropy、grad_norm 一起看。

#### `main_actor/kl_loss`、`reranker_actor/kl_loss`

actor 训练时的 KL regularization loss，用来约束当前策略不要偏离参考策略太远。它不是 `algorithm.use_kl_in_reward` 里的 reward penalty，而是 actor loss 里的 KL 项。

#### `main_actor/ppo_kl`、`reranker_actor/ppo_kl`

PPO 统计出的策略 KL 指标，用来观察新旧策略或参考策略之间的偏移程度。当前日志里多为 0 或很小，说明短 run 中策略变化不大。

#### `main_actor/entropy`、`reranker_actor/entropy`

策略输出分布的熵。越高表示输出分布越分散，探索更强；越低表示策略更确定。

#### `main_actor/grad_norm`、`reranker_actor/grad_norm`

actor 更新时的梯度范数，用来观察训练稳定性。过大可能意味着梯度爆炸或不稳定；长期为 0 则可能说明没有有效更新。

#### `main_actor/lr`、`reranker_actor/lr`

当前学习率。

### 长度 / 行为指标

#### `main_response_length/mean`

main agent 平均 response token 长度。

#### `main_response_length/clip_ratio`

main agent 输出达到 `max_response_length` 上限的比例。这个值高说明不少样本被截断，可能影响答案质量和 reward。

#### `main_num_turns/mean`

main agent 平均对话轮数。CoSearch 中轮数越高通常意味着搜索/工具调用更多。

#### `reranker_response_length/mean`

reranker 输出平均 token 长度。

#### `reranker_response_length/clip_ratio`

reranker 输出达到最大 response 长度的比例。

#### `reranker_num_turns/mean`

reranker agent 的平均轮数。当前 reranker 通常是固定工具响应/单轮结构，所以这个指标变化较小。

### Advantage / Return 指标

#### `main_critic/advantages/mean`、`reranker_critic/advantages/mean`

PPO/GRPO 计算出的 advantage 均值。它表示某个 token/action 相对同组样本或基线的好坏。由于 advantage 常被归一化，均值不一定能直接解释为“模型变好/变差”。

#### `main_critic/returns/mean`、`reranker_critic/returns/mean`

用于训练 value/策略目标的 return。当前训练更关注 actor，具体意义要结合 advantage estimator。

### 性能 / 资源指标

#### `perf/time_per_step`

单个 train step 总耗时，包含 rollout、reward、logprob、actor update、dump/save 等。

#### `perf/throughput`

训练吞吐，通常按 token/time 统计。越高越好，但会受 response 长度、搜索耗时、reranker 数据量影响。

#### `timing_per_token_ms/gen`

生成阶段平均每 token 耗时，主要反映 rollout 生成效率。

#### `main_perf/mfu/actor`、`reranker_perf/mfu/actor`

MFU，即 Model FLOPs Utilization，表示模型训练计算对理论峰值算力的利用率估计。它比 `nvidia-smi` 的 GPU util 更偏向训练计算效率，但不直接等于整卡利用率。

#### `main_perf/max_memory_allocated_gb`、`reranker_perf/max_memory_allocated_gb`

PyTorch 记录的最大已分配显存。

#### `main_perf/max_memory_reserved_gb`、`reranker_perf/max_memory_reserved_gb`

PyTorch caching allocator 预留的最大显存，通常会高于或等于 allocated。

#### `main_perf/cpu_memory_used_gb`、`reranker_perf/cpu_memory_used_gb`

进程视角记录的 CPU 内存使用量。

### Reranker 样本构造 / 过滤指标

#### `reranker_pre_filter/total_outputs`

过滤前构造出来的 reranker rollout 样本数。

#### `reranker_post_filter/total_outputs`

过滤后保留下来的 reranker 样本数。

#### `reranker_cap/final_total_outputs`

经过 cap/sampling 后最终用于 reranker 训练的样本数。

#### `reranker_filter/filter_ratio`

被过滤掉的比例。

#### `reranker_duplicate/ratio`

重复样本比例。

#### `reranker_pre_filter/ratio_valid`

过滤前 reranker 样本中 reward 非负、格式/执行有效的比例。

#### `reranker_pre_filter/avg_agent_score`

reranker 输出后，下游 main/search-r1 agent 继续回答得到的平均 agent reward。

#### `reranker_pre_filter/avg_tool_score`

reranker tool 自身的平均 reward。

#### `reranker_pre_filter/avg_final_score`

聚合后的 reranker final reward 平均值。

## Q3. 此模型训练过程中，产出最终回复时产生的梯度也被计算和更新了吗？

会。当前训练中，最终回复阶段的 assistant 生成 token 会参与 PPO actor loss 的计算，因此会产生梯度并更新模型参数。

更准确地说，CoSearch 多轮 rollout 中有三类 token：

1. prompt / 用户问题 token
2. assistant 生成 token，包括中间 `<reason>...</reason><tool_call>...</tool_call>` 和最终 `<reason>...</reason><answer>...</answer>`
3. tool observation token，也就是 retrieval/reranker 返回给模型看的内容

代码里的 `response_mask` 区分哪些 token 参与训练：

```text
response_mask = 1：LLM 自己生成的 token
response_mask = 0：tool response / observation / padding token
```

实现位置：

- `CoSearch/verl/verl/experimental/agent_loop/agent_loop.py`
  - `generate_sequences` 的注释明确说明：`response_mask` 对 LLM generated tokens 为 1，对 observation/padding 为 0。
  - 多轮场景下，前一段 assistant tool call 和最后 assistant answer 都属于 LLM generated tokens。
- `CoSearch/verl/verl/workers/actor/dp_actor.py`
  - actor update 调用 policy loss 时传入 `response_mask=response_mask`。
- `CoSearch/verl/verl/workers/roles/utils/losses.py`
  - `policy_loss_fn(..., response_mask=response_mask, ...)`，说明 PPO loss 只在 `response_mask=1` 的 token 上聚合。

因此：

```text
最终 <answer>...</answer> 的生成 token：参与梯度更新
中间 <tool_call>...</tool_call> 的生成 token：参与梯度更新
retrieval/reranker 返回的 tool observation token：不参与梯度更新
padding token：不参与梯度更新
```

当前脚本还设置了：

```text
algorithm.adv_estimator=grpo
```

GRPO 的实现会先把每条 rollout 的 sequence-level reward 求和成一个标量 score，再在同一 uid 组内做相对 advantage，然后把这个标量 advantage broadcast 到所有 `response_mask=1` 的 token 上：

```text
advantages = scalar_advantage.unsqueeze(-1) * response_mask
```

所以最终答案不是只在最后一个 token 上更新。虽然 reward 本身被写在最后一个有效 response token 上，但经过 GRPO advantage 后，整条由模型生成的 response 轨迹都会收到同一个 outcome-level 信号，包括最终回复 token 和前面的工具调用 token。

需要注意的是，这里说的“最终回复过程产生梯度”指的是训练阶段 actor update 重新计算 logprob 并反传。rollout 生成本身是采样/推理过程，不在生成时直接反传；梯度是在后续 `update_actor` 阶段基于 rollout 数据计算出来的。

## Q4. 训练过程如果每次重启都只训练 100 个 step，实际用于训练的数据是同样的，还是从全训练集中随机采样的？

当前 09b 这类训练脚本下，不是固定取全训练集前 100 个 step 对应的数据；它会从完整训练集上用随机 sampler 打乱顺序。因此每次重启只训练 100 step 时，实际用到的样本通常不是同一批。

当前这次 run 的配置和日志可以确认几点：

```text
TRAIN_DATA = data/co_search/local_flashrag/co_search_rl_51k.train.parquet
TRAIN_MAX_SAMPLES = 51200
data.train_batch_size = 64
data.shuffle = True
data.seed = None
algorithm rollout_n = 8
dataset len = 51200
```

这里要区分两个层面。

第一，训练样本池是否固定：

`TRAIN_MAX_SAMPLES=51200`，而日志里的 `dataset len: 51200`，说明当前没有从全训练集里额外截取一个更小子集。也就是说，训练样本池就是这 51200 条完整训练数据。

第二，每次只跑 100 step 时，取到哪 100 个 batch：

VERL 的 `create_rl_sampler` 在 `data.shuffle=True` 时使用 `RandomSampler`。当前 `data.seed=None`，所以每次训练进程启动时随机顺序不固定。100 step 实际上会消费随机排列后的前 100 个 batch。

以当前 `TRAIN_BATCH_SIZE=64` 计算：

```text
100 step 约消耗 100 * 64 = 6400 条原始 prompt 样本
```

同时因为 `rollout_n=8`，每个 prompt 会生成 8 条 rollout 轨迹，所以训练日志里一个 step 会看到：

```text
64 prompts * 8 rollouts = 512 trajectories
```

这解释了日志里 `batch_size=512` 或 `trajectories=512/512` 的来源：它不是 512 条不同原始问题，而是 64 条 prompt 各采样 8 次。

因此，当前默认行为是：

```text
每次重启训练：
  使用同一个 51200 条训练集池
  但随机打乱顺序
  只跑 100 step 时，取随机顺序中的前 6400 条 prompt
  因 seed=None，不保证和上一次相同
```

如果希望每次 100 step 都严格用同一批数据，有两种做法：

1. 设置固定 seed，例如通过 Hydra 覆盖 `data.seed=某个固定整数`，同时保持 `data.shuffle=True`。
2. 关闭 shuffle，即设置 `data.shuffle=False`，这样会按 parquet 顺序从前往后取数据。

更推荐第一种，因为它既保留随机打乱，又能保证实验可复现。

## Q5. 当前 09 和 09b 脚本使用了测试集吗？它们使用了验证集吗？

结论：

```text
09/09b 默认不使用测试集训练。
09/09b 默认配置了验证集路径，但默认不会实际跑验证评估。
```

### 是否使用测试集？

没有。09 和 09b 的默认训练数据是：

```text
data/co_search/local_flashrag/co_search_rl_51k.train.parquet
```

它们没有把 `data/co_search/local_flashrag/eval_by_dataset/*.parquet` 这些测试/评测集作为训练输入，也没有默认读取完整评测集：

```text
data/co_search/local_flashrag/co_search_7bench.eval.parquet
```

所以默认训练不会用测试集。

### 是否配置了验证集？

配置了。09 和 09b 都会 export：

```text
VAL_DATA = data/co_search/local_flashrag/co_search_7bench_smoke.eval.parquet
VAL_MAX_SAMPLES = 8
VAL_BATCH_SIZE = 8
```

底层 `train_cosearch_verl_base.sh` 也会把它传给 VERL：

```text
data.val_files = [VAL_DATA]
data.val_max_samples = VAL_MAX_SAMPLES
data.val_batch_size = VAL_BATCH_SIZE
```

但这只是“配置了 val_files”，不等于训练时实际执行 validation。

### 默认是否实际跑验证？

默认不跑。09/09b 都设置：

```text
TEST_FREQ = -1
```

底层 03 脚本还显式设置：

```text
trainer.val_before_train = False
trainer.val_only = False
trainer.test_freq = TEST_FREQ
```

所以默认情况下：

- 训练前不跑 validation。
- 训练中不按 step 周期跑 validation。
- 训练结束后也不会得到真实 validation metrics。

当前 10-step run 的日志也验证了这一点：

```text
test_freq: -1
```
